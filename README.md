# ecoFAST plots repository

This repository temporarily hosts precomputed plot images used by the ecoFAST Shiny application.

The plots are currently stored on GitHub and accessed remotely by the app.  
In the future, plots may be migrated to a more permanent storage solution.

It contains:

- classification plots;
- abrupt-shift detection plots;
- dynamical footprint plots;
- plot manifests used by the app for indexing and filtering;
- helper scripts used to generate plots.

The repository is designed to remain separate from:

- the main **ecoFAST** R package;
- the **ecoFAST_app** Shiny repository.

This separation avoids storing large image files inside the package or app repositories.

---

# Repository structure

Recommended structure:

```text
ecoFAST_plots/
├── template_dataset_manifest.r
├── create_manifest_fishbiomass_aicasd.r
├── fishbiomass/
│   └── aicasd/
│       ├── classif/
│       │   └── *.png
│       ├── detection/
│       │   └── *.png
│       ├── dynfoot/
│       │   └── *.png
│       ├── manifest.csv
│       └── produce_plots.r
└── README.md
```

---

# Plot types

The repository can contain three categories of plots:

| Folder | Description |
|---|---|
| `classif/` | Classification plots generated by `run_classif()` |
| `detection/` | Abrupt-shift detection plots generated by `run_classif()` |
| `dynfoot/` | Dynamical footprint plots generated by `run_dynfoot()` |

The `detection/` folder is optional and may not exist for some workflows, such as standard `aic` analyses.

---

# Manifest files

Each dataset-analysis combination should contain a `manifest.csv` file.

The manifest indexes all available plots and links:

- dataset name;
- analysis type;
- time-series ID;
- plot type;
- plot file path.

The ecoFAST Shiny app uses this file to:

- quickly retrieve plots;
- filter plots by time series;
- avoid scanning image folders dynamically.

---

# Manifest generation

Manifest files are generated from:

```text
template_dataset_manifest.r
```

For each dataset and analysis:

1. copy the template script;
2. rename it appropriately;
3. edit only lines 2 and 3.

Example:

```r
dataset <- "fishbiomass"
analysis <- "aicasd"
```

Recommended filename:

```text
create_manifest_<dataset>_<analysis>.r
```

Run the script to generate:

```text
<dataset>/<analysis>/manifest.csv
```

The script automatically detects which plot folders exist and includes only available plot types.

---

# Producing plots

Each dataset-analysis folder may contain a helper script such as:

```text
produce_plots.r
```

used to generate all plots for that dataset.

Typical workflow:

1. run ecoFAST analyses with produce_plots.r;
2. save plots into:
   - `classif/`
   - `detection/`
   - `dynfoot/`
3. run the manifest creation script.

# Naming conventions

## Analysis folders

Recommended analysis names:

- `aic`
- `aicasd`
- `loo`

---

## Plot files

Plot filenames should preserve the original time-series ID whenever possible.

Examples:

```text
ACADRED2J3K_aicasd_looFALSE_allmodels.png
COD1F-XIV_aicasd_looTRUE_allmodels.png
```

---

# Integration with the ecoFAST app

The Shiny app reads plot manifests from this repository.

Dataset metadata and plot URLs are configured in:

```text
config/datasets.csv
```

inside the `ecoFAST_app` repository.

The app can then load plots from a remote static file server.

---

# Adding a new dataset

Example workflow for adding a new dataset named `ram` using an `aic` workflow:

```text
ecoFAST_plots/
└── ram/
    └── aic/
        ├── classif/
        ├── dynfoot/
        ├── manifest.csv
        └── produce_plots.r
```

Steps:

1. generate ecoFAST outputs;
2. generate plots;
3. create `manifest.csv`;
4. register the dataset in `ecoFAST_app/config/datasets.csv`.

---

# Requirements

The repository assumes that:

- ecoFAST is installed;
- plots were generated using ecoFAST functions;
- manifests were created using `template_dataset_manifest.r`.

---

# Notes

This repository only stores plots and plot metadata. Analytical outputs (`.rda` objects) are stored separately in the `ecoFAST_app` repository.

This repository is currently a temporary hosting solution. The manifest system was designed to remain compatible with future storage backends.
